Caso Práctico: Eligiendo el Modelo de Servicio Correcto

Aplica tus conocimientos para guiar a InnovaTech Solutions en su viaje a la nube.

El Desafío

Eres el nuevo Arquitecto de Soluciones en **InnovaTech Solutions**, una empresa de software en crecimiento. Tu primera tarea es asesorar a diferentes equipos sobre la mejor estrategia para desplegar tres nuevas iniciativas en la nube. Para cada escenario, debes analizar los requisitos, considerar las compensaciones y recomendar el modelo de servicio (IaaS, PaaS o SaaS) más adecuado. Utiliza el **Modelo de Responsabilidad Compartida** y el **Marco de Decisión Estratégico** para justificar tus decisiones.

Escenario 1: "El Legado" - Migración del ERP Crítico

El departamento de finanzas utiliza un sistema de Planificación de Recursos Empresariales (ERP) que ha funcionado durante 10 años en un servidor físico en la oficina. El hardware está obsoleto y falla constantemente. El software del ERP requiere una versión muy específica y antigua de Red Hat Enterprise Linux y una base de datos Oracle 10g. El proveedor del ERP ya no existe y no hay presupuesto para modernizar el software, solo para moverlo ("lift-and-shift") a un lugar más fiable.

Reflexión del Arquitecto:

  • ¿Cuál es la prioridad principal aquí: control total del entorno, velocidad de despliegue o facilidad de uso?
  • ¿Qué tan grande es la carga operativa que el equipo puede asumir según el modelo de responsabilidad? ¿Necesitan acceso al SO?
  • ¿Se requiere una personalización profunda a nivel de sistema operativo y base de datos para garantizar la compatibilidad?

¿Qué modelo de servicio recomendarías?

Escenario 2: "El Nuevo Producto" - Lanzamiento de App Web Moderna

Un equipo de desarrollo ágil ha construido una nueva aplicación web para clientes usando Node.js y una base de datos PostgreSQL. La principal prioridad del negocio es lanzar la aplicación al mercado lo más rápido posible para obtener feedback de los primeros usuarios. El equipo de desarrollo es pequeño y quiere centrarse exclusivamente en escribir código y añadir nuevas funcionalidades, no en gestionar servidores, parches de seguridad o configurar el escalado.

Reflexión del Arquitecto:

  • Según el marco de decisión, ¿qué es más importante: la velocidad de salida al mercado o la flexibilidad total?
  • ¿El equipo quiere asumir una carga operativa alta o baja?
  • ¿La aplicación utiliza tecnologías estándar (Node.js, PostgreSQL) que una plataforma gestionada podría soportar fácilmente?

¿Qué modelo de servicio recomendarías?

Escenario 3: "La Herramienta Interna" - Un CRM para Ventas

El equipo de ventas ha crecido rápidamente y necesita urgentemente una herramienta de Gestión de Relaciones con Clientes (CRM) para gestionar sus contactos y oportunidades. Sus necesidades son estándar: seguimiento de clientes, gestión de pipeline de ventas y reportes básicos. No hay desarrolladores disponibles para construir una solución personalizada y la necesitan funcionando "para ayer". El presupuesto es operativo (OPEX), no de capital (CAPEX).

Reflexión del Arquitecto:

  • ¿La necesidad es consumir un software o construir uno nuevo?
  • ¿Cuál es el requisito de velocidad de salida al mercado? ¿Se puede esperar a un ciclo de desarrollo?
  • ¿Quién debe ser responsable del mantenimiento del software, según el modelo de responsabilidad?

¿Qué modelo de servicio recomendarías?
